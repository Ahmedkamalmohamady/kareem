# وثيقة متطلبات النظام (SRS)

## 1. المقدمة
### 1.1. الغرض
توضيح الهدف من تطوير تطبيق إدارة التصاريح للبوابات.

### 1.2. نطاق المشروع
تقديم وصف مختصر للتطبيق ووظائفه.

## 2. وصف النظام
### 2.1. المستخدمون
- **الشركة**: تقدم طلبات التصريح.
- **مدير شركة الأمن**: يراجع الطلبات ويقبلها أو يعدلها.
- **السائق**: يستقبل QR code ويدخل من البوابة.
- **رجل الأمن**: يتحقق من التصاريح عند البوابة.

### 2.2. الوظائف الأساسية
1. **طلب تصريح**: إدخال بيانات التصريح (اسم السائق، رقم السيارة، البوابة، تاريخ الدخول).
2. **مراجعة الطلب**: قبول أو تعديل الطلب من قبل مدير شركة الأمن.
3. **إصدار QR code**: إرسال QR code للسائق في حال قبول الطلب.
4. **التحقق من التصريح**: مسح QR code من قبل رجل الأمن عند البوابة.
5. **تسجيل الزيارات**: تسجيل دخول السيارة في سجل الزيارات.

## 3. المتطلبات الوظيفية
### 3.1. متطلبات واجهة المستخدم
- واجهة سهلة الاستخدام لكل من الشركة ومدير الأمن والسائق ورجل الأمن.

### 3.2. متطلبات النظام
- قاعدة بيانات لتخزين بيانات التصاريح، المستخدمين، وسجل الزيارات.
- نظام لتنبيه المستخدمين بحالة الطلبات (قبول/رفض).

## 4. المتطلبات غير الوظيفية
### 4.1. الأداء
- يجب أن يكون النظام قادرًا على معالجة الطلبات بشكل سريع.

### 4.2. الأمان
- تأمين البيانات الشخصية وحماية QR code من التلاعب.

## 5. السيناريوهات
### 5.1. سيناريو طلب تصريح
1. الشركة تدخل البيانات.
2. ترسل الطلب إلى مدير الأمن.
3. مدير الأمن يقبل أو يعدل الطلب.
4. إذا تم القبول، يُرسل QR code للسائق.

### 5.2. سيناريو دخول السائق
1. السائق يدخل البوابة مع QR code.
2. رجل الأمن يمسح QR code.
3. إذا كان التصريح صالحًا، يسمح بالدخول ويُسجل الزيارة.

## 6. القيود
- يجب أن تتوافق جميع العمليات مع الأنظمة الحالية للأمن.
